O corpo de categorize: ordem importa, e o compilador só te cobre até certo ponto

A lição mais valiosa do curso: descobrir onde os tipos param de te proteger.

Win de hoje

Você escreve o corpo de categorize com a regra "tests primeiro" (§4.2 do PRD), entende o erro "nem todo caminho retorna um valor", e — o mais importante — sente na pele a fronteira: o compilador garante que você devolve algum Layer, nunca que é o certo. Esse é o trabalho dos testes.

Na Lição 0001 você tipou a casca: categorize(path: string): Layer. Hoje preenchemos o miolo — uma sequência de regras, cada uma checando o caminho e devolvendo um literal de camada.

1. A regra: primeira que casa vence

function categorize(path: string): Layer {
  if (path.endsWith("Test.php"))            return "tests";
  if (path.includes("/Actions/"))          return "action";
  if (path.includes("database/migrations/")) return "migration";
  if (path.includes("/Models/"))            return "model";
  return "outros";                          // balde final
}
Buraco de JS preenchido path.endsWith("Test.php") e path.includes("/Actions/") são métodos de string do próprio JavaScript (não são TS). endsWith → termina com? · includes → contém? Ambos devolvem boolean. O TS só acrescenta saber que path é string, então ele oferece esses métodos no autocomplete e barra path.toFixed() (que é de número).

2. A garantia que o compilador dá: todo caminho retorna um Layer

Apague o return "outros" final e o TypeScript recusa o código: 

// Function lacks ending return statement and return type
// does not include 'undefined'.  (ts 2366)

Ele analisou todos os caminhos do if e achou um furo: se nenhum casar, a função devolveria undefined — que não é um Layer. Em JS puro isso passaria batido e estouraria só lá na frente, quando alguém usasse o resultado. O balde "outros" (regra 4 do §4.1 do PRD) existe justamente para fechar esse furo.

3. A garantia que o compilador não dá: a camada certa

Olhe este caminho real — um teste que vive sob uma pasta Actions:

tests/Feature/Actions/CreateUserActionTest.php

Ele casa com duas regras:

endsWith("Test.php") → tests includes("/Actions/") → action

Como "primeira que casa vence", a ordem decide. Se você checasse Actions antes de tests, esse arquivo de teste seria rotulado "action" — errado. E aqui está o ponto da lição: o compilador fica calado. Tanto "action" quanto "tests" são Layer válidos; a assinatura está respeitada. O tipo garante a forma, não a lógica. É por isso que o PRD manda checar tests primeiro.

A lição mental do dia Tipos provam formato ("é um Layer"). Testes provam comportamento ("é o Layer certo para este caminho"). Você precisa dos dois. Um não substitui o outro.

4. Provar a lógica com um teste (na Playground)

Ainda não temos o app nem um test runner — isso vem na issue 01. Mas dá para provar a lógica agora com uma asserção que roda no navegador: 

function expectLayer(path: string, esperado: Layer) {
  const got = categorize(path);
  console.log(got === esperado ? "✓" : "✗ ESPERAVA " + esperado, path, "→", got);
}

expectLayer("tests/Feature/Actions/CreateUserActionTest.php", "tests");
expectLayer("app/Contexts/Crm/Actions/CreateUser.php",         "action");
expectLayer("database/migrations/2024_create_plans_table.php",   "migration");
expectLayer("composer.json",                                     "outros");
Ponte com PHP É o mesmo instinto de um teste PHPUnit (assertEquals($esperado, categorize($path))), só que aqui o parâmetro esperado: Layer também é checado pelo compilador: se você escrever expectLayer("…", "actoin") no teste, o typo nem compila.

Pratique (recuperação ativa)

Responda de memória antes de revelar.

P1 Você apaga o return "outros" final. O que acontece?

O TS analisa todos os caminhos: sem o balde final, existe um caso (nenhum if casa) em que a função devolveria undefined — que não é Layer. Erro de compilação (ts 2366), antes de rodar. Essa é a garantia que ele dá.

P2 Com a regra de /Actions/ checada antes da de Test.php, o caminho tests/Feature/Actions/...ActionTest.php é classificado como…

"Primeira que casa vence": se Actions vem antes, ele retorna "action" — e está errado (devia ser tests). O compilador não reclama porque "action" é um Layer legítimo. Só um teste pega esse bug. Por isso: tests primeiro.

P3 Quando categorize compila sem erro, o tipo Layer garante que…

Tipos garantem formato: toda saída é um dos 16 literais. Eles não sabem qual é o correto para um caminho, nem se sua ordem de regras faz sentido — isso é lógica, e lógica se prova com teste.

Faça agora (5 minutos)

  1. Na Playground, junte o type Layer da 0001 + o categorize + o bloco de expectLayer acima.
  2. Abra o console (aba "Logs" da Playground) e confira que os 4 casos dão ✓.
  3. Quebre de propósito: mova a regra de /Actions/ para antes da de Test.php. Rode de novo. Qual caso vira ✗? Esse é o bug que a ordem do PRD evita.
  4. Desfaça e confirme que voltou tudo a ✓.

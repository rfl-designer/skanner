Skanner · TypeScript · Lição 0001

O tipo Layer: uma de 16, e nem uma a mais

Seu primeiro tipo de verdade — e o recurso que faz o TypeScript valer a pena.

Win de hoje

Ao final você escreve, sozinho, a assinatura function categorize(path: string): Layer e entende por que o compilador recusa "migraton" (com o typo) antes de o app rodar — algo que o PHP e o JavaScript só descobririam quebrando em produção.

O coração do Skanner (§4.2 do PRD) é pegar o caminho de cada arquivo da PR e dizer a qual camada ele pertence — e são exatamente 16 camadas canônicas, nesta ordem fixa:

migrationmodelenumdtopolicyactionservicejobeventlistenerobservernotificationlivewirebladetestsoutros

"Uma de um conjunto fixo de valores" é exatamente o que TypeScript modela melhor que qualquer linguagem que você já usou. Vamos chegar lá em três passos.

1. TypeScript é JavaScript + uma camada de tipos que some

Você escreve TS, uma ferramenta checa os tipos e depois os apaga, gerando JavaScript comum que o Electron roda. Os tipos existem só em tempo de escrita/compilação; em runtime não sobra nada deles. [handbook]

Ponte com PHP Em function categorize(string $path): Layer, o PHP confere o type hint quando a função roda — erro vira exceção em runtime. No TS é o oposto: a checagem acontece antes de rodar e some depois. Você troca "quebra em produção" por "sublinhado vermelho no editor".

2. Literal type: um tipo que é um único valor

Você já viu string (qualquer texto). A virada do TypeScript: um tipo pode ser um valor exato. O tipo "migration" aceita só a string "migration" — nada mais. [handbook] 

let camada: "migration";   // tipo = o literal "migration"
camada = "migration";        // ok
camada = "model";            // ✗ erro: "model" não é "migration"

Um literal sozinho é inútil. O poder vem de combinar vários.

3. Union: uma de várias — o tipo Layer

A barra | é "ou". Uma union de literais diz "o valor é um destes". Damos um nome a ela com type (um type alias): 

type Layer =
  | "migration" | "model" | "enum" | "dto"
  | "policy" | "action" | "service" | "job"
  | "event" | "listener" | "observer" | "notification"
  | "livewire" | "blade" | "tests" | "outros";
Ponte com PHP Parece um enum Layer: string do PHP 8.1 — e a intenção é a mesma: um conjunto fechado de valores nomeados. Mas a union de TS é mais leve: é só um tipo, não uma classe; some em runtime; e qualquer string "action" literal já é um Layer, sem Layer::Action->value.

4. A assinatura tipada de categorize

Agora a função-coração ganha um contrato que o compilador faz cumprir: entra string (o caminho), sai Layer (nunca outra coisa). 

function categorize(path: string): Layer {
  if (path.includes("database/migrations/")) return "migration";
  if (path.includes("/Models/"))            return "model";
  return "outros";
}

Por que isso muda seu dia a dia, comparado a JS puro:

  • Typo vira erro na hora. return "migraton"; não compila — não é um Layer.
  • Autocomplete das 16 opções. Ao digitar return ", o editor lista só camadas válidas.
  • Quem usa o resultado fica seguro. Um switch (camada) sabe que só existem esses 16 casos.
Detalhe que confunde no começo A ordem das pílulas lá em cima (migration → … → outros) é a ordem de exibição do review, não algo que a union imponha. Uma union não tem ordem; ela só diz quais valores valem. Ordenar as camadas é assunto de outra lição.

Pratique (recuperação ativa)

Responda de memória antes de conferir — o esforço de lembrar é o que fixa.

P1 Quando o código TypeScript roda como JavaScript no Electron, os tipos…

TS checa os tipos em tempo de compilação e depois os apaga: o que roda é JS comum, sem rastro de tipo. Por isso o erro aparece no editor, não em produção.

P2 Com type Layer = "migration" | "model" | "tests", o compilador aceita return "migraton" (com o typo) dentro de categorize?

O retorno foi tipado como Layer, que só admite os três literais. "migraton" não está na lista → erro de compilação. É justamente o bug que o tipo existe para pegar.

P3 Para dizer "o valor é uma de migration ou model", qual declaração está certa?

| é "ou" (union): uma de várias — o que queremos. & seria "e ao mesmo tempo" (impossível para dois literais diferentes: nenhuma string é as duas). Os [colchetes] seriam uma tupla com os dois valores juntos, não um único valor.

Faça agora (5 minutos, sem instalar nada)

  1. Abra a TypeScript Playground.
  2. Cole o tipo Layer completo (as 16) e a função categorize do passo 4.
  3. Troque um return por "migraton" e veja o sublinhado vermelho aparecer.
  4. Apague o return "outros" final e leia o erro — o compilador vai reclamar que "nem todo caminho retorna um Layer". Pense por que ele tem razão.

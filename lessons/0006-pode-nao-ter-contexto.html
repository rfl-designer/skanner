Skanner · TypeScript · Lição 0006

Pode não ter contexto: string | null e a checagem que o compilador exige

Onde uma função admite "não sei" como resposta — e o TS te obriga a tratar esse caso.

Win de hoje

Você escreve a assinatura de resolveContext (PRD §4.1): dado um arquivo, qual feature (contexto) ele pertence? Diferente de categorize, a resposta pode ser nenhuma — o balde "Sem contexto". Isso é string | null, e a grande sacada de hoje: com null no tipo, o compilador não te deixa usar o valor sem antes checar se ele existe.

O categorize (0002) sempre tinha resposta — o balde "outros" fechava o furo, então o retorno era um Layer cheio, sempre. resolveContext é diferente: muitos arquivos (config, CI, uma migration órfã) não têm feature, e o PRD manda jogá-los num balde "Sem contexto". A função precisa poder dizer "não achei" — e o tipo tem que refletir isso.

1. Por que string, e não uma union como Layer?

As camadas são um conjunto fechado: as 16 são conhecidas de antemão → union de literais (0001). Já os contextos são um conjunto aberto: Crm, Billing, Activities… qualquer pasta nova em app/Contexts/ é um contexto válido, e você não tem como listá-los todos antes. Conjunto aberto de textos → string.

A régua: union ou string? Conjunto fechado e conhecido (camadas, perfil "modular" | "flat") → union de literais, e o compilador cobra os valores. Conjunto aberto (nome de contexto, título da PR) → string. Escolher errado aqui é o erro de modelagem nº 1 de quem vem de linguagens sem union.

2. A assinatura: o retorno admite ausência

function resolveContext(path: string): string | null {
  // devolve o nome do contexto, ou null quando não há feature ("Sem contexto")
}

string | null é uma union (0001) — "uma de: um texto, ou o valor null". null é o jeito do JS de dizer "vazio de propósito, ausência de valor". Pôr null no tipo é uma promessa honesta: "essa função às vezes não tem resposta, e eu te avisei".

Ponte com PHP É o tipo nullable do PHP: function resolveContext(string $path): ?string. O ?string do PHP é exatamente string | null do TS. A diferença grande vem no passo 3: o TS te força a tratar o null antes de usar o valor — o PHP deixa você esquecer e explodir em runtime com "Call to a member function on null".

3. O win: o compilador não deixa você ignorar o null

Com strictNullChecks ligado (o padrão na Playground), o resultado de resolveContext não é uma string usável de cara — é "string ou null", e o TS exige que você decida o que fazer no caso vazio: 

const ctx = resolveContext(file.path);

console.log(ctx.toUpperCase());
// ✗ 'ctx' is possibly 'null'. (null não tem .toUpperCase → o app quebraria)

Você estreita o tipo tratando o caso vazio — o mesmo narrowing da 0002, agora sobre null:

const ctx = resolveContext(file.path);

if (ctx === null) {
  // arquivo vai para o balde "Sem contexto" (PRD §4.1, regra 4)
} else {
  // aqui o TS SABE que ctx é string — pode usar ctx.toUpperCase() sem medo
  console.log(ctx.toUpperCase());
}

Dentro do else, o TS removeu o null do tipo: ele provou, lendo o seu if, que ali só sobra string. O "possivelmente null" some sozinho. É o compilador te empurrando para tratar o caso vazio antes de rodar — o bug de null morto na escrita.

4. Um corpo de partida (regra 1: path direto)

Para a função não ficar vazia, a primeira regra do §4.1 — path em app/Contexts/<Ctx>/:

function resolveContext(path: string): string | null {
  const marker = "app/Contexts/";
  if (path.includes(marker)) {
    const resto = path.split(marker)[1];   // "Crm/Actions/CreateUser.php"
    return resto.split("/")[0];            // "Crm"
  }
  return null;                              // ainda não resolveu → "Sem contexto" por ora
}
Buraco de JS preenchido — .split() "a/b/c".split("/") quebra a string nos separadores e devolve um array: ["a", "b", "c"] — é o explode("/", $s) do PHP. Pegando o [0] você fica com o primeiro pedaço. Aqui: corta no "app/Contexts/", pega o que vem depois, e o primeiro segmento é o nome do contexto.

Falta as regras 2–4 (path de teste, ponte por nome, balde) — e a regra 3 precisa de uma peça nova: o conjunto de contextos já achados na PR. Isso é a 0007 (e traz o tipo Set). Hoje o que importa é o tipo de retorno honesto e a checagem de null.

Pratique (recuperação ativa)

Responda de memória antes de revelar.

P1 Por que o contexto é string e a camada era uma union de literais (Layer)?

Union de literais serve quando você conhece todos os valores de antemão (as 16 camadas). Contextos surgem livremente (Crm, Billing, …), conjunto aberto → string.

P2 Uma função retorna string | null. Você pode chamar resultado.toUpperCase() direto?

Com strictNullChecks (padrão na Playground), o valor é "string ou null", e null não tem .toUpperCase(). O compilador exige que você trate o vazio antes — o bug morre na escrita, não em runtime (diferente do PHP).

P3 Depois de if (ctx === null) { … } else { … }, qual o tipo de ctx dentro do else?

Isso é narrowing (0002): o TS lê o fluxo e, no else, sabe que null já foi descartado — sobra string. Por isso lá dentro o .toUpperCase() é aceito sem reclamação.

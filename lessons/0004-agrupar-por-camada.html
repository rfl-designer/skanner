Agrupar por camada: Record<Layer, ChangedFile[]>

Onde a lista plana da 0003 vira a árvore que a navegação lateral da tela de review percorre.

Win de hoje

Você pega a ChangedFile[] plana (0003) e a indexa por camada — um balde de arquivos para cada Layer. O tipo disso é Record<Layer, ChangedFile[]>: sua primeira estrutura aninhada, e exatamente o que a navegação Feature → Camada → Arquivo do PRD (§6) consome.

A PR vira uma lista plana de ChangedFile — ótimo para guardar, péssimo para exibir. A tela não mostra "47 arquivos em ordem"; ela mostra seções por camada: as migrations juntas, depois os models juntos, e assim por diante. Para isso o app precisa de baldes: cada Layer apontando para os seus arquivos.

1. Record<K, V>: um dicionário tipado

type GroupedByLayer = Record<Layer, ChangedFile[]>;

Leia em voz alta: "para cada Layer (a chave K), uma ChangedFile[] (o valor V)". Record é o jeito do TS de descrever um objeto-dicionário: você diz o tipo das chaves e o tipo dos valores, de uma vez.

Ponte com PHP É o array associativo do PHP — array<string, ChangedFile[]> — mas com uma trava que o PHP não te dá: aqui a chave não é uma string qualquer, é um Layer. Tentar usar groups["modle"] é erro de compilação, igual a "modle" ter sido recusado lá na 0001. A union Layer continua trabalhando por você — agora como conjunto de chaves válidas.

2. O compilador exige todas as camadas

Esta é a parte que conecta com a sua pergunta sobre exaustividade. Como a chave é Layer (as 16 camadas), um objeto desse tipo precisa ter as 16 chaves — faltou uma, erro: 

const groups: GroupedByLayer = {
  migration: [],
  model: [],
  action: [],
  tests: [],
  // … e as outras 12. Se você parar aqui:
};
// ✗ Property 'enums' is missing in type '{ migration: …; }'
//   but required in type 'Record<Layer, ChangedFile[]>'.

O mesmo mecanismo que recusou "modle" agora garante que nenhuma camada fique de fora. Amanhã você adiciona uma camada nova ao Layer — e o compilador acende em todo Record<Layer, …> que esqueceu de tratá-la. Exaustividade de graça: o tipo não te deixa esquecer um caso.

3. Construir os baldes: for…of

Com os baldes vazios prontos, basta percorrer os arquivos e empurrar cada um no balde da sua camada: 

for (const file of files) {
  groups[file.layer].push(file);
}
// groups.migration → [todas as migrations], groups.model → [todos os models], …

Repare em groups[file.layer]: file.layer é um Layer (veio da interface ChangedFile da 0003), então o TS sabe que essa chave existe e que o valor é uma ChangedFile[] — por isso o .push(file) é aceito sem reclamação e sem checagem de undefined. A inicialização do passo 2 (todos os baldes como []) é o que torna isso seguro: nunca há balde faltando.

Buraco de JS preenchido — for…of for (const x of lista) percorre os valores de um array, um por vez — é o foreach ($lista as $x) do PHP. (Cuidado: existe também for…in, que percorre chaves/índices — não é o que queremos aqui.) E .push(item) adiciona no fim do array, como $arr[] = $item. Em PHP o todo seria $groups[$file->layer][] = $file; — a diferença é que aqui o compilador confere a chave.

4. Por que "aninhada", e o que vem depois

Você acabou de montar uma estrutura de dois níveis: um objeto cujos valores são, eles mesmos, arrays de objetos (Layer → ChangedFile[]). Esse é o caso do repo flat do PRD (§4): só Camada → [arquivos], sem feature.

No repo modular, a árvore ganha mais um andar por cima — Feature → Camada → [arquivos]. A forma disso é só envolver o de hoje: Record<string, Record<Layer, ChangedFile[]>> (feature → baldes por camada). É para onde a 0005 vai. Hoje você fincou o nível de dentro.

Pratique (recuperação ativa)

Responda de memória antes de revelar.

P1 O que Record<Layer, ChangedFile[]> descreve?

Record<K, V> = "para cada chave do tipo K, um valor do tipo V". Aqui K = Layer e V = ChangedFile[] → um dicionário camada → arquivos daquela camada. É o balde por camada que a tela percorre.

P2 Num Record<Layer, ChangedFile[]>, o que acontece se o objeto literal esquecer uma das camadas?

Como a chave é a union Layer, o tipo exige todas as camadas presentes. Faltou uma → "Property '…' is missing". É a mesma checagem que recusa um valor fora da union — agora garantindo exaustividade: nenhuma camada fica esquecida.

P3 Por que groups[file.layer].push(file) não exige checar se o balde existe?

file.layer tem tipo Layer (veio da interface ChangedFile), e o Record<Layer, …> garante um valor para todo Layer. Logo o TS sabe que groups[file.layer] é uma ChangedFile[] de verdade — nunca undefined. A inicialização com baldes vazios é o que sustenta essa garantia.

Faça agora (5 minutos)

  1. Na Playground, parta do que você já tem (Layer + categorize + interface ChangedFile + a lista files da 0003).
  2. Crie const groups: Record<Layer, ChangedFile[]> = { … } com todas as camadas inicializadas em []. Depois o for…of empurrando cada arquivo.
  3. Provoque a exaustividade: apague uma das chaves do literal e leia o erro "Property '…' is missing". Devolva a chave e o erro some.
  4. Tente groups["modle"] — veja o TS recusar uma chave que não é um Layer.
  5. console.log(groups) e confira na aba Logs que cada camada tem os arquivos certos (e as vazias ficaram []).

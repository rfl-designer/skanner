Ordenar para exibir: do índice à lista que a tela percorre

Onde você descobre que o tipo diz quais camadas existem — mas não em que ordem mostrá-las.

Win de hoje

Você transforma o Record<Layer, ChangedFile[]> da 0004 na lista ordenada e sem camadas vazias que a tela de review renderiza: uma LayerGroup[] na ordem fixa do PRD (§4.2). E entende a verdade por trás disso: ordem é um dado que você declara, não algo que o tipo garante.

O groups da 0004 é ótimo para buscar ("me dá os arquivos de model") — mas péssimo para exibir. A tela precisa das camadas numa ordem fixa (migration primeiro, tests perto do fim) e sem mostrar as vazias (PRD §4.2). E aqui mora a armadilha: um objeto/Record não promete ordem nenhuma das chaves. Confiar nela para renderizar é bug na certa.

1. A verdade central: tipo não é ordem

Record<Layer, ChangedFile[]> diz quais chaves existem (as 16 camadas) — e nada sobre a ordem de exibição. Ordem de exibição é informação do domínio (o PRD decidiu migration → … → tests → outros), então você a declara como dado: uma lista. 

const LAYER_ORDER: Layer[] = [
  "migration", "model", "enums", /* … as 16, na ordem do §4.2 … */
  "action", "tests", "outros",
];
// Layer[] = "uma lista de Layer" (0003). Cada item é checado:
// um "modle" aqui = erro, igual na 0001.
Ponte com PHP Em PHP é a diferença entre um array associativo (['model' => [...]], bom pra buscar pela chave) e um array sequencial (['migration', 'model', ...], que guarda a ordem). É a mesma divisão aqui: o Record indexa; a lista LAYER_ORDER ordena. Cada um para seu trabalho.
Detalhe honesto O compilador confere que cada item de LAYER_ORDER é um Layer válido — mas não exige que você liste todos os 16 (diferente do literal do Record na 0004, que exigia todas as chaves). Esqueceu uma camada na ordem? Ela simplesmente não aparece na tela, sem erro. Vale conferir na mão — ou, mais à frente, derivar uma da outra. Por hoje, declare as 16 com cuidado.

2. Montar a lista ordenada: .map sobre a ordem

Em vez de percorrer as chaves do Record (cuja ordem não é confiável), você percorre a sua lista e busca cada balde por nome: 

interface LayerGroup {
  layer: Layer;
  files: ChangedFile[];
}

const ordered: LayerGroup[] = LAYER_ORDER.map((layer) => ({
  layer: layer,
  files: groups[layer],
}));
// ordered[0] = { layer: "migration", files: [...] }, na ordem que VOCÊ definiu.

Saiu de um Record (sem ordem) para um LayerGroup[] (com ordem) — porque array preserva a sequência dos itens. Os (parênteses) em volta do { } são os mesmos da 0003: sem eles o { viraria bloco de código, não objeto.

3. Omitir as vazias: .filter

Camadas que a PR não tocou ficaram com []. O PRD manda escondê-las. Filtra: 

const visible: LayerGroup[] = ordered.filter((group) => group.files.length > 0);
// só os grupos que têm pelo menos 1 arquivo. É isso que a UI renderiza.
Buraco de JS preenchido — .filter() .filter(fn) percorre o array e devolve um novo array só com os itens em que fn deu true (o original não muda — irmão do .map da 0003). É o array_filter() do PHP, ou ->filter() das Collections do Laravel. .length é o tamanho do array (como count()); > 0 = "tem algo".

4. Onde isso encaixa no app

visible: LayerGroup[] é exatamente o que a tela de review percorre: a navegação lateral Camada → Arquivo (PRD §6) faz um .map sobre essa lista para desenhar uma seção por camada, na ordem certa, sem vazias. No React vai ser literalmente visible.map(group => …) — por isso valeu a pena terminar num array, não num Record.

Falta um andar: quem é a Feature de cada arquivo (repo modular, PRD §4.1). Isso é uma função nova de peso — resolveContext, com 4 regras de prioridade — e ganha a 0006. Depois dela, a árvore Feature → Camada → Arquivo fecha.

Pratique (recuperação ativa)

Responda de memória antes de revelar.

P1 Um Record<Layer, ChangedFile[]> garante a ordem em que as camadas serão exibidas?

Objeto/Record descreve quais chaves existem, não a ordem de exibição. Para mostrar numa sequência fixa você declara essa ordem como dado (LAYER_ORDER: Layer[]) e percorre ela.

P2 O que ordered.filter(g => g.files.length > 0) devolve?

.filter devolve um novo array mantendo só os itens em que a função deu true; o original fica intacto. Aqui: fora as camadas com files vazio — cumprindo o "camadas vazias são omitidas" do PRD §4.2.

P3 Por que terminar num LayerGroup[] (array) em vez de entregar o Record direto para a UI?

A UI desenha as camadas em sequência, então precisa de algo ordenado e iterável em ordem — um array. O Record é a etapa intermediária boa para indexar; o array ordenado é a forma boa para exibir.

Faça agora (5 minutos)

  1. Na Playground, parta do groups que você montou na 0004.
  2. Declare const LAYER_ORDER: Layer[] = [ … ] na ordem do PRD §4.2.
  3. Monte ordered: LayerGroup[] com o .map sobre LAYER_ORDER, depois visible com o .filter tirando as vazias. console.log(visible).
  4. Provoque: ponha um "modle" no LAYER_ORDER e leia o erro (item fora da union Layer).
  5. Veja a diferença: comente o .filter e confira que as camadas vazias voltam a aparecer em ordered — é o filtro que cumpre o §4.2.
Fonte primária de hoje: MDN — Array.prototype.filter() (e .map() ao lado). O conceito de tipo é o da 0003 (T[] + interface); hoje a novidade é a ideia de ordem-como-dado, não um tipo novo.
